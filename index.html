TP PCA / PRA — Infrastructure AD

1. Introduction

Ce TP a pour objectif de mettre en place une infrastructure réseau comprenant un contrôleur de domaine, un serveur de sauvegarde et un serveur web. Il permet également de comprendre les notions de PCA (Plan de Continuité d'Activité) et PRA (Plan de Reprise d'Activité).

2. Objectifs

  • Mettre en place un serveur Active Directory
  • Configurer les services DNS et DHCP
  • Créer des utilisateurs
  • Tester la connectivité avec un client
  • Comprendre le fonctionnement du PCA/PRA

3. Mise en place de l'infrastructure

Création de la machine virtuelle SRV-AD sous Windows Server.
Configuration du réseau en mode réseau interne.
Configuration de l'adresse IP statique 172.16.1.10.

4. Installation Active Directory

Installation du rôle AD-Domain-Services via PowerShell.
Création du domaine btssio.local.
Redémarrage du serveur après promotion.

5. Configuration DNS et DHCP

Vérification du DNS.
Installation du rôle DHCP.
Création de l'étendue d'adresses IP : 172.16.1.50 à 172.16.1.150.
Configuration de la passerelle et du DNS.

6. Création des utilisateurs

Création d'un utilisateur test (Jean Dupont).
Attribution d'un mot de passe.
Vérification de la création avec une commande PowerShell.

7. Tests

Test du fonctionnement du serveur avec un client.
Vérification de l'attribution d'une adresse IP.
Connexion avec l'utilisateur créé.

8. Conclusion

Ce TP a permis de mettre en place une infrastructure complète avec Active Directory, DNS et DHCP. Le serveur fonctionne correctement et permet de gérer les utilisateurs et les machines du réseau. Cependant, le PRA reste limité car il n'y a pas de redondance matérielle.

Évaluation Docker — Réseaux — BTSSIO-02 — 25-26

Étape 1 : Création des réseaux

Création de trois réseaux virtuels totalement isolés pour définir nos zones de sécurité.

docker network create --subnet 172.30.0.0/24 wan_net
docker network create --subnet 172.20.0.0/24 dmz_net
docker network create --subnet 172.10.0.0/24 lan_net

Étape 2 : Configuration du Pare-feu

Déploiement d'un conteneur central (passerelle) connecté aux trois réseaux. Configuration d'iptables pour bloquer tout le trafic par défaut et autoriser uniquement le flux HTTP vers la DMZ et le NAT pour le LAN.

docker run -dit --name firewall --cap-add NET_ADMIN --cap-add NET_RAW \
  --sysctl net.ipv4.ip_forward=1 --network wan_net alpine
docker network connect dmz_net firewall
docker network connect lan_net firewall
docker exec -it firewall sh
apk update && apk add iptables iproute2
iptables -F
iptables -P FORWARD DROP
iptables -A FORWARD -i eth0 -o eth1 -p tcp --dport 80 -j ACCEPT
iptables -A FORWARD -i eth2 -o eth1 -p tcp --dport 80 -j ACCEPT
iptables -A FORWARD -i eth2 -o eth0 -j ACCEPT
iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE

Étape 3 : Serveur Web (DMZ)

docker run -dit --name web_dmz --cap-add NET_ADMIN --network dmz_net nginx
docker exec -it web_dmz sh
apt update && apt install -y iproute2
ip route add 172.10.0.0/24 via 172.20.0.2
ip route add 172.30.0.0/24 via 172.20.0.2

Étape 4 : Déploiement des Clients

Création de deux machines de test (interne et externe). Installation de curl et définition de leur route par défaut vers le pare-feu.

Client LAN

docker run -dit --name lan_client --cap-add NET_ADMIN --network lan_net alpine
docker exec -it lan_client sh
apk update && apk add iproute2 curl
ip route add 172.20.0.0/24 via 172.10.0.2

Client WAN

docker run -dit --name wan_client --cap-add NET_ADMIN --network wan_net alpine
docker exec -it wan_client sh
apk update && apk add iproute2 curl
ip route add 172.20.0.0/24 via 172.30.0.2

Étape 5 : Tests et Validation

Validation de l'architecture. Les requêtes HTTP traversent le pare-feu et le serveur web répond correctement aux deux zones.

docker exec -it lan_client curl http://172.20.0.3
docker exec -it wan_client curl http://172.20.0.3

Projet : Sécurisation du site WordPress monchauffeurspl.fr et mise en place d'un suivi de carburant

Introduction

La sécurité informatique est un enjeu majeur pour les entreprises, en particulier lorsqu'elles utilisent un site web pour communiquer avec leurs clients. Un site non sécurisé peut exposer des données sensibles, nuire à l'image de l'entreprise et interrompre l'activité.

Dans le cadre de mon alternance, j'ai été chargé de sécuriser le site WordPress de l'entreprise et d'améliorer certains outils internes, notamment la gestion du carburant. Ce projet a donc deux objectifs :

  • Sécuriser le site web contre les menaces
  • Mettre en place un outil de suivi du carburant fiable

Chapitre 1 — Présentation de l'entreprise

GRAND TRANSPORT OUVERT est une entreprise spécialisée dans le transport de marchandises. Elle utilise un site WordPress pour présenter ses services et communiquer avec ses clients. L'environnement technique comprend :

  • Hébergement OVH
  • CMS WordPress
  • Plugins (WooCommerce, Elementor)

Avant mon arrivée, aucune politique de sécurité n'était réellement mise en place.

Chapitre 2 — Analyse du besoin et cahier des charges

Problèmes identifiés — Site web

  • Absence de certificat SSL → échanges non sécurisés
  • Plugins obsolètes → failles de sécurité
  • Spam massif → dégradation du site (20 commentaires suspects Bitcoin/crypto en attente)
  • Pas de sauvegarde → risque de perte totale

Problèmes identifiés — Gestion interne

  • Aucun suivi du carburant
  • Difficulté à détecter les anomalies
  • Risque de vol de carburant

Besoins

  • Sécuriser le site web et protéger les données
  • Mettre en place une stratégie de sauvegarde
  • Créer un outil de suivi du carburant automatisé

Chapitre 3 — Choix techniques et architecture

Pour répondre aux problèmes de spam massif et de vulnérabilités, j'ai opté pour une stratégie de défense en plusieurs couches :

  • Mise à jour de l'environnement : Passage à la dernière version de WordPress et des plugins pour corriger les failles critiques.
  • Antispam Bee : Plugin gratuit, conforme au RGPD, qui filtre les commentaires sans utiliser de Captcha intrusif.
  • WP Armour : Technique du Honeypot (pot de miel) pour bloquer les robots de spam de manière invisible.
  • Filtrage natif : Utilisation des outils internes de WordPress pour le blocage par mots-clés et adresses IP.

Chapitre 4 — Conception détaillée

Audit technique

J'ai utilisé l'outil "Santé du site" pour analyser l'état des plugins, la version PHP et les problèmes de sécurité. Résultats : PHP obsolète, plugins non à jour, absence de SSL.

Plan de sécurisation

  • Mise à jour globale
  • Installation anti-spam
  • Mise en place de sauvegardes
  • Préparation HTTPS

Chapitre 5 — Réalisation

5.1. Sécurisation multicouche du site

Étape 1 — Filtrage natif WordPress : Validation manuelle obligatoire du premier commentaire. Liste noire de mots-clés (btc, crypto, deposit, viagra...). Bannissement d'IP malveillantes (ex : 2.58.56.93, plages Tor 185.220.101.x).
Étape 2 — Antispam Bee : Blocage des commentaires selon la langue (Français uniquement). Suppression automatique des spams après 5 jours.
Étape 3 — WP Armour (Honeypot) : Champ invisible que seuls les robots remplissent → identification et blocage instantané avant soumission.
Étape 4 — Sauvegardes : Stratégie de sauvegarde régulière incluant les fichiers du site et la base de données SQL. Continuité de l'activité garantie.

Résultat obtenu : Cette double approche (automatique avec les plugins et manuelle avec les filtres WordPress) a permis d'éliminer la quasi-totalité des messages indésirables.

5.2. Mise en place du suivi de carburant (Excel / VBA)

Objectif : Mettre en place un outil permettant de suivre la consommation de carburant des véhicules, analyser les dépenses et automatiser le traitement des données.

  • Import des données AS24 (CSV/Excel) — Portail de gestion de flotte
  • Nettoyage des données (doublons, formats dates/montants)
  • Organisation par véhicule, date et type de dépense

Automatisation avec Excel :

  • Formules SI, RECHERCHEV, INDEX/EQUIV
  • Calcul automatique de la consommation (litres / kilomètres)
  • Mise en forme conditionnelle pour détecter les anomalies

Analyse des données :

  • Tableaux croisés dynamiques (TCD)
  • Analyse de la consommation par véhicule
  • Comparaison des dépenses par période mensuelle

Automatisation avec VBA :

  • Formulaire pour faciliter la saisie des données
  • Enregistrement automatique dans le tableau
  • Génération de rapports de suivi

Résultats : Gain de temps dans le traitement des données, réduction des erreurs humaines, meilleure visibilité sur les dépenses de carburant, aide à la détection d'anomalies ou de surconsommation.

Chapitre 6 — Validation et recette

  • Site web : Vérification des mises à jour ✓ — Test anti-spam ✓ — Test sauvegarde ✓
  • Excel : Vérification des calculs ✓ — Tests des automatisations ✓ — Validation par l'entreprise ✓

Chapitre 7 — Bilan personnel

Compétences développées : Cybersécurité WordPress, analyse de vulnérabilités, automatisation Excel / VBA.

Difficultés rencontrées : Compréhension des outils de sécurité, mise en place des automatisations.

Chapitre 8 — Perspectives

  • Activation complète du HTTPS : Finaliser l'installation du certificat SSL via l'hébergeur OVH.
  • Mise à jour de l'environnement serveur : Migration vers PHP 8.3.
  • Monitoring : Mise en place d'un outil de surveillance pour être alerté en temps réel.

Conclusion

Ce projet m'a permis de sécuriser un site web réel et de mettre en place un outil de gestion utile pour l'entreprise. Il m'a permis de développer des compétences techniques en cybersécurité et en gestion de données.
